[Room Database Singleton 초기화와 thread-safe 처리]

이 프로젝트의 AppDatabase는 전역에서 하나만 사용되는 singleton 객체로 관리했습니다.
현재 프로젝트 규모만 보면 여러 스레드가 동시에 DB 인스턴스를 생성할 가능성은 높지 않습니다.
다만 database 초기화 코드는 한 번만 어긋나도 재현과 디버깅이 어려운 영역이기 때문에,
비용이 크지 않은 범위에서 안전성을 확보하는 방향으로 구현했습니다.

이를 위해 INSTANCE에는 @Volatile을 적용했고, getDatabase() 내부에서는 synchronized를 사용했습니다.

- @Volatile
  다른 스레드가 INSTANCE의 최신 값을 일관되게 볼 수 있도록 하여,
  singleton 객체가 생성된 뒤 안전하게 공유되도록 돕습니다.

- synchronized
  동시에 여러 실행 흐름이 getDatabase()에 진입하더라도
  AppDatabase가 중복 생성되지 않도록 초기화 구간을 보호합니다.

즉, 이 코드는 “현재 반드시 멀티스레드 경쟁이 발생한다”는 가정보다,
전역 singleton 초기화 지점은 안전하게 닫아두는 편이 낫다는 판단에 가깝습니다.
성능 비용도 database가 최초 생성되는 구간에 제한적이므로,
프로젝트 전체에서 과도한 오버엔지니어링이라고 보지는 않았습니다.

추후 Hilt 같은 DI를 도입하면 database 생성 책임을 DI 컨테이너로 옮겨
이 초기화 코드를 더 단순하게 정리할 수 있습니다.
하지만 DI를 사용하지 않는 현재 구조에서는,
명확한 의도를 가진 방어적 singleton 초기화 방식으로 설명할 수 있습니다.

면접 설명 한 줄 요약:
“현재 프로젝트에서 동시 초기화 가능성은 높지 않았지만,
AppDatabase는 전역 singleton이라 초기화 안정성이 중요하다고 판단해
@Volatile과 synchronized를 사용해 thread-safe 하게 구성했습니다.”
