==================================================
Room Database Singleton 초기화와 Thread Safety
==================================================

이 문서는 AppDatabase의 singleton 초기화 코드에서
왜 @Volatile과 synchronized를 함께 사용했는지 정리한 학습 노트입니다.

--------------------------------------------------
1. 이 노트의 핵심 주제
--------------------------------------------------

AppDatabase는 전역에서 하나만 사용되는 singleton 객체로 관리합니다.
현재 프로젝트 규모만 보면 여러 스레드가 동시에 DB 인스턴스를 생성할 가능성은 높지 않습니다.
하지만 database 초기화 코드는 한 번만 어긋나도 재현과 디버깅이 어려운 영역이기 때문에,
비용이 크지 않은 범위에서 안전성을 확보하는 방향으로 구현했습니다.

--------------------------------------------------
2. @Volatile의 의미
--------------------------------------------------

@Volatile은 다른 스레드가 INSTANCE의 최신 값을 일관되게 볼 수 있도록 도와줍니다.
즉, singleton 객체가 생성된 뒤 안전하게 공유되도록 가시성을 확보하는 장치입니다.

간단히 기억하면:
- 누군가가 INSTANCE를 바꿨을 때
- 다른 실행 흐름도 그 최신 값을 보게 하려는 장치

--------------------------------------------------
3. synchronized의 의미
--------------------------------------------------

synchronized는 동시에 여러 실행 흐름이 getDatabase()에 들어오더라도,
초기화 구간을 한 번에 하나만 통과하게 만듭니다.

즉, AppDatabase가 중복 생성되지 않도록
초기화 구간을 보호하는 역할입니다.

--------------------------------------------------
4. 왜 둘 다 사용했는가
--------------------------------------------------

역할이 다르기 때문입니다.

- @Volatile → 최신 값이 잘 보이게 함
- synchronized → 초기화 경쟁 자체를 막음

즉, 하나는 가시성 문제에 가깝고,
하나는 동시 진입 제어 문제에 가깝습니다.

--------------------------------------------------
5. 이 선택이 과한가
--------------------------------------------------

현재 프로젝트만 보면 실제로 멀티스레드 경쟁이 자주 발생하지 않을 수 있습니다.
그럼에도 이 코드를 유지하는 이유는 다음과 같습니다.

- DB singleton 초기화는 실패 비용이 큼
- 코드 비용은 크지 않음
- 의도가 분명한 방어 코드임

즉,
"반드시 경쟁이 발생할 것"이라는 가정보다,
"전역 singleton 초기화는 안전하게 닫아두는 편이 낫다"는 판단에 가깝습니다.

--------------------------------------------------
6. 추후 구조가 바뀌면 어떻게 될까
--------------------------------------------------

추후 Hilt 같은 DI를 도입하면
database 생성 책임을 DI 컨테이너로 옮겨 이 초기화 코드를 더 단순하게 정리할 수 있습니다.
하지만 DI를 사용하지 않는 현재 구조에서는,
명확한 의도를 가진 방어적 singleton 초기화 방식으로 설명할 수 있습니다.

--------------------------------------------------
7. 이 노트에서 특히 기억할 핵심 4문장
--------------------------------------------------

1) AppDatabase는 전역 singleton이므로 초기화 안정성이 중요하다.
2) @Volatile은 최신 INSTANCE 값의 가시성을 높인다.
3) synchronized는 동시에 여러 초기화가 겹치지 않게 막는다.
4) 현재 구조에서는 비용이 적은 안전장치로 볼 수 있다.

--------------------------------------------------
8. 면접 또는 설명용 한 줄 요약
--------------------------------------------------

"현재 프로젝트에서 동시 초기화 가능성은 높지 않았지만, AppDatabase는 전역 singleton이라 초기화 안정성이 중요하다고 판단해 @Volatile과 synchronized를 사용해 thread-safe 하게 구성했습니다."
