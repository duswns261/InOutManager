==================================================
Application과 AppContainer 이해하기
==================================================

이 문서는 InOutMangerApplication.kt를 기준으로,
Application 클래스가 왜 필요한지와 왜 AppContainer를 여기서 생성하는지 정리한 학습 노트입니다.

--------------------------------------------------
1. 이 파일의 역할을 한 줄로 정리
--------------------------------------------------

InOutMangerApplication은
"앱이 시작될 때 전역 준비물을 한 번 세팅해 두는 진입 클래스"입니다.

즉, 이 파일의 핵심 책임은 화면을 그리는 것이 아니라
앱 전체에서 공통으로 사용할 의존성을 준비하는 것입니다.

--------------------------------------------------
2. 왜 Application이 필요한가
--------------------------------------------------

안드로이드 앱은 보통 다음 순서로 준비됩니다.

1) 앱 프로세스 생성
2) Application 객체 생성
3) Application.onCreate() 실행
4) Activity 생성
5) MainActivity.onCreate() 실행

즉, Application은 Activity보다 먼저 생성됩니다.
그래서 Database, Repository, AppContainer처럼
여러 화면에서 공유되고 비교적 오래 살아야 하는 객체를 준비하기에 적합합니다.

--------------------------------------------------
3. 현재 코드에서 실제로 하는 일
--------------------------------------------------

현재 코드는 onCreate()에서 아래 작업을 합니다.

- DefaultAppContainer(this) 생성
- container 프로퍼티에 보관

이 의미는 다음과 같습니다.

- 앱 시작 시점에 전역 의존성 보관함을 준비한다.
- 이후 Activity는 필요할 때 application.container를 통해 접근한다.
- MainActivity가 Database와 Repository 생성 방법까지 알 필요가 없어진다.

즉, 생성 책임을 Activity 밖으로 밀어낸 구조입니다.

--------------------------------------------------
4. 왜 this를 넘기는가
--------------------------------------------------

여기서 this는 InOutMangerApplication 객체 자신입니다.
Application 역시 Context 계열이므로 Room Database 생성에 필요한 Context를 전달할 수 있습니다.

중요한 점은
이 Context가 특정 화면에 묶인 Activity Context가 아니라,
앱 전역과 연결된 Application Context 성격이라는 점입니다.
그래서 전역 객체 생성에 더 적절합니다.

--------------------------------------------------
5. Activity Context와 비교하면 왜 더 적합한가
--------------------------------------------------

Activity Context는 특정 화면 생명주기에 묶입니다.
그래서 오래 살아야 하는 객체가 Activity Context를 오래 잡고 있으면
메모리 누수 위험이 생길 수 있습니다.

반면 Application은 프로세스 단위에서 비교적 오래 유지되므로,
전역 객체 초기화 지점으로 더 안전한 선택입니다.

정리하면:
- 화면/테마/UI 관련 작업 → Activity Context가 더 적합한 경우가 많음
- DB/Repository/전역 객체 준비 → Application Context가 더 적합함

--------------------------------------------------
6. 이 구조가 왜 좋은가
--------------------------------------------------

이 구조의 핵심 장점은 책임 분리입니다.

- Application: 전역 준비물 초기화
- AppContainer: 의존성 생성 및 보관
- Activity: ViewModel 획득과 UI 시작

이렇게 나누면 MainActivity가 가벼워지고,
구조를 읽기도 쉬워집니다.
또한 Hilt 같은 DI 프레임워크를 배우기 전,
수동 DI 흐름을 직접 이해하기에 좋은 구조입니다.

--------------------------------------------------
7. 이 파일에서 특히 기억할 핵심 4문장
--------------------------------------------------

1) Application은 Activity보다 먼저 생성되므로 전역 준비물을 두기 좋다.
2) InOutMangerApplication은 AppContainer를 소유하는 전역 진입점이다.
3) 오래 살아야 하는 객체 생성에는 Application Context가 더 적합하다.
4) 이 구조의 목적은 Activity에서 생성 책임을 걷어내는 것이다.

--------------------------------------------------
8. 면접 또는 설명용 한 줄 요약
--------------------------------------------------

"Application 클래스에서 AppContainer를 초기화해 앱 전역 의존성의 생성 책임을 UI 레이어 밖으로 분리했고, MainActivity는 ViewModel 획득과 화면 구성에 더 집중하도록 정리했습니다."
