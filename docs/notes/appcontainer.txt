==================================================
AppContainer와 수동 DI 구조 이해하기
==================================================

이 문서는 AppContainer.kt를 기준으로,
왜 인터페이스와 DefaultAppContainer 구조를 두었는지 정리한 학습 노트입니다.
그리고 수동 DI 관점에서 어떤 의미가 있는지 정리한 학습 노트입니다.

--------------------------------------------------
1. 이 파일의 역할을 한 줄로 정리
--------------------------------------------------

AppContainer는
"앱 전역에서 공유할 의존성을 생성·보관·제공하는 수동 DI 컨테이너"입니다.

즉, 단순한 보관함이 아니라
필요한 의존성을 한곳에 모아두는 생성 진입점입니다.

--------------------------------------------------
2. AppContainer 인터페이스를 왜 두는가
--------------------------------------------------

현재 코드는 아래 구조를 사용합니다.

- AppContainer → 계약
- DefaultAppContainer → 실제 구현

이렇게 나누면 외부에서는
"무엇을 제공하는가"만 알면 되고,
내부에서 실제로 어떻게 만들었는지는 몰라도 됩니다.

즉,
- 계약과 구현 분리
- 테스트 대체 가능성 확보
- 구조 설명이 더 명확해짐
- 향후 구현 교체 포인트를 확보

이 구조의 의미는
"무엇을 제공할지"와 "어떻게 만들지"를 분리하는 것입니다.

- AppContainer → 계약(제공 항목)
- DefaultAppContainer → 기본 구현(실제 생성 방식)

즉, 외부에서는 productRepository를 받기만 하면 되고
내부에서 Room으로 만들었는지, Fake로 만들었는지는 몰라도 됩니다.

--------------------------------------------------
3. DefaultAppContainer는 실제로 무엇을 하는가
--------------------------------------------------

DefaultAppContainer는 Context를 받아,
AppDatabase를 준비하고,
거기서 productDao()를 꺼내 ProductRepository를 만듭니다.

흐름:
Context
→ AppDatabase.getDatabase(context)
→ productDao()
→ ProductRepository(...)
→ productRepository 제공

즉, 실제 생성 코드는 여기 모여 있습니다.

--------------------------------------------------
4. by lazy를 쓴 이유
--------------------------------------------------

현재 코드는 database와 productRepository 모두 lazy를 사용합니다.

의미:
- AppContainer가 생성되는 순간 바로 만들지 않음
- 실제로 처음 필요할 때 생성
- 이후에는 같은 인스턴스를 재사용
- 배포할 앱에서 초기 실행 화면을 만들것이기 때문에 lazy로 생성하도록 설정 유지

장점:
1) 앱 시작 시 초기 비용을 조금 늦출 수 있음
2) 아직 쓰지 않는 객체를 미리 만들지 않음
3) 전역 공유 객체이면서도 불필요한 초기화를 줄일 수 있음

--------------------------------------------------
5. Context를 여기에 두는 것은 괜찮은가
--------------------------------------------------

현재 구조는 Application에서 전달된 Context를 사용합니다.
이 경우 전역 객체 초기화 용도로는 적절한 선택입니다.

중요한 점은
Activity Context를 오래 붙잡고 있지 않는다는 것입니다.
지금처럼 Application에서 생성되고 Room Database 준비에 활용되는 구조는 비교적 안전합니다.

--------------------------------------------------
6. 수동 DI 관점에서 어떻게 이해하면 좋은가
--------------------------------------------------

지금 구조를 수동 DI 흐름으로 보면 다음과 같습니다.

Application
→ DefaultAppContainer 생성

DefaultAppContainer
→ ProductRepository 생성

MainActivity
→ AppContainer에서 ProductRepository 획득

InventoryViewModel
→ 제공받은 ProductRepository 사용

즉, 객체가 필요한 곳이 직접 new 하지 않고
바깥에서 준비된 것을 전달받는 구조입니다.
이것이 바로 DI의 핵심입니다.

--------------------------------------------------
7. 이 파일에서 특히 기억할 핵심 5문장
--------------------------------------------------

1) AppContainer는 앱 전역 의존성을 생성·보관·제공하는 수동 DI 컨테이너다.
2) 인터페이스와 구현을 나누면 계약과 생성 방식을 분리할 수 있다.
3) DefaultAppContainer는 실제로 Database와 Repository를 연결해 만든다.
4) lazy를 사용해 필요한 시점에 의존성을 지연 생성한다.
5) 이 구조는 Hilt를 배우기 전 DI 흐름을 이해하는 데 좋다.

--------------------------------------------------
8. 면접 또는 설명용 한 줄 요약
--------------------------------------------------

"AppContainer 인터페이스와 DefaultAppContainer 구현을 두어 앱 전역 의존성의 생성·보관 책임을 한곳으로 모았고, lazy 초기화를 통해 필요 시점에 Database와 Repository를 준비하는 수동 DI 구조로 정리했습니다."
